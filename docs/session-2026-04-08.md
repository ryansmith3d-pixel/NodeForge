# Idiograph — Session Summary
**Date:** April 8, 2026
**Status:** FROZEN — historical record, do not revise
**Session type:** Design

## Context

This session focused entirely on demo interface design — no implementation. Three areas were
covered in sequence: competitive landscape survey, verification of two core critique observations,
and color philosophy for the demo UI. A significant new build target emerged: a standalone Qt
color design tool with a live SSE connection to the D3 demo interface.

---

## Competitive Landscape — What Was Surveyed

Free tools reviewed in prior sessions by the user. This session focused on freemium and paid tools:

- **Scite** ($12–20/month) — citation context analysis (supports / contradicts / mentions). Most
  interesting as a thesis foil: typed edge semantics implemented as an NLP layer on top of an
  opaque system. The right insight, the wrong architecture.
- **Litmaps** (freemium) — timeline-based layout (temporal axis + relatedness axis). Strongest
  spatial encoding decision in the category. Design reference for the temporal view.
- **ResearchRabbit** (free/freemium) — best visual UI in the category. Interaction model worth
  studying for what projection decisions were made implicitly.
- **Elicit** (freemium) — not a graph tool, but the mature version of structured paper extraction.
  Useful as contrast to graph-based approach.
- **VOSviewer** — academic/enterprise, designed for bibliometricians. Does not hide complexity.
  Useful precedent for the idea that transparency is an audience choice, not a technical
  limitation.

**Key assessment:** Premium tools are well-designed and demonstrate genuine data science depth.
Idiograph is not competing with them. The demo is evidence for an argument, not a product pitch.

## Observation 1 — AI-Personalized Search: Verified

**Claim:** Multiple tools are inserting AI personalization into search in ways that are a
category error for research.

**Verified.** R Discovery explicitly advertises personalized reading feeds based on reading habits.
ResearchRabbit offers AI-suggested papers based on saved collections. Semantic Scholar delivers
personalized research feeds based on user interests.

**Refined distinction:** There are two magnitudes of the problem:
- **Personalized recommendations** (here are papers you might like) — wrong philosophy for
  research, where the goal is discovery of the unknown, not reinforcement of prior interest.
- **Personalized search results** (same query returns different results for different users) —
  worse, because it silently breaks the assumption that a query is a query. R Discovery appears
  to do this.

**Idiograph's counter-position:** Same query, same graph, every time. Deterministic. This is not
a missing feature — it is a deliberate architectural choice.

**Essay note:** The personalization critique belongs in the essay but should not name specific
tools. The demo makes the argument affirmatively. The contrast is implicit; the viewer supplies it.

## Observation 2 — Transparency / Black Box: Verified and Sharpened

**Claim:** These tools hide their methods behind black box proprietary techniques.

**Verdict:** Partially right, but the framing needed precision. The sharper claim is stronger.

**What they do document:** Connected Papers documents its technique category — co-citation and
bibliographic coupling. That is a known, named bibliometric method.

**Where the black box actually lives:** Semantic Scholar uses a learned embedding model trained
with contrastive learning. You know the technique category but cannot inspect why a specific edge
between two specific papers exists. Algorithm documented at system level; opaque at instance level.

Connected Papers confirmed directly: *"Everything is proprietary for now."* ResearchRabbit's
algorithm is described by UCL library guides as simply "not clear." An MIT Press paper on citation
graph architecture states the core problem precisely: the edge tells you that p1 cites p2 but
"does not specify the why or the how."

**The precise and correct critique:**

> You can know what class of algorithm produced a graph. You cannot ask "why is this specific
> edge here, with this weight, connecting these two nodes." That question has no answerable
> surface in any of these tools.

This is not a documentation problem. It is an architectural one. The answer does not exist because
the structure that would produce it was never built.

**Idiograph's counter-position:** Instance-level traceability. Every edge has a reason that can
be inspected. This is the thesis applied to the visual layer.

**Demo implication:** Every UI element either shows data or shows how the data was produced.
Nothing decorative. Nothing that obscures the decision chain. Click a node, see not just what it
is but what path through the graph produced it.

## Color Philosophy — Established

**Base environment:**
- Dark but not black. Warm dark gray, not cold. Gaea v2 (version 1.2 Enterprise) is the closest
  tonal reference.
- Muted and desaturated throughout. Low ambient contrast.
- High contrast and punchy color are reserved for where the user should look.

**Red discipline:**
- Red avoided entirely except for genuine error states.
- A technical audience reads red as pipeline failure. Decorative red destroys the signal.

**Semantic encoding principle:**
- Every color choice maps to a named, inspectable semantic role.
- Color encodes meaning, not decoration.
- The mapping is always explicit and auditable — consistent with the thesis.

**Edge and port layer:**
- Informed by Gaea and Softimage ICE reference images provided by the user.
- In production node graph tools, color lives primarily on edges and ports, not node bodies.
- Node bodies stay near-neutral — slightly lighter than canvas, just enough to read as distinct.
- Edge color carries semantic load: encode by edge type (citation, co-citation, bibliographic
  coupling, execution dependency).
- Port color matches edge color — the connection is visually traceable end to end.
- Selection and highlight state is the one moment for a punchy accent color.
- Canvas is the darkest surface in the hierarchy.

**Status:** Color philosophy established. Specific hue decisions are an ongoing process, expected
to evolve as real graph density is applied. Architecture supports live iteration via the color
design tool (see below).

## New Build Target — Qt Color Design Tool

**What it is:** A standalone PySide6 desktop application for designing and iterating the Idiograph
interface color palette. Separate from the Idiograph core. Does not touch the graph system.

**Output:** Named semantic token file (JSON). Roles map to hex values. Example structure:
```json
{
  "surface.base": "#1e1e2e",
  "node.default": "#4a4a6a",
  "node.highlight": "#7eb8f7",
  "edge.citation": "#f7a948",
  "edge.cocitation": "#4ab8f7",
  "alert": "#f7c948",
  "error": "#c0392b"
}
```

**Live connection architecture:**
- Qt tool posts token updates to a FastAPI endpoint on value change.
- D3 frontend holds an open SSE (Server-Sent Events) connection to FastAPI.
- Token update broadcasts immediately; D3 re-renders without page reload.
- No file watching, no polling, no rebuild cycle.

**Architectural significance:**
- The color tool is just another client of the FastAPI layer — same principle as MCP.
- Every interface is a view. The color designer is a view into token state; the graph is a view
  into execution state.
- The live parameter channel between two discrete apps is the "coring" pattern from production
  VFX tooling (Rhythm & Hues and others used socket-based cross-app parameter driving ~15–20
  years ago). SSE over HTTP is the modern equivalent — same concept, cleaner plumbing.

**Thesis connection:**
- The color tool's output port connects to the demo's style input port.
- This is the thesis running as tooling infrastructure, not just as a demo subject.
- Explicit, typed connections between discrete systems. Inspectable at any point.

**Implementation notes:**
- PySide6 (not PyQt6) — LGPL licensed, Qt6, actively maintained.
- Scope is self-contained: pure UI tool producing a JSON file. Does not touch Idiograph core.
- Good candidate for a focused implementation session.
